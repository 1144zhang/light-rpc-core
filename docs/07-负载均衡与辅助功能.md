# 负载均衡与辅助功能详解

## 一、负载均衡

### 1.1 为什么需要负载均衡

注册中心返回的服务列表有多个节点，选哪个？

```
服务列表: [192.168.1.10:8080, 192.168.1.11:8080, 192.168.1.12:8080]
               ↑              ↑              ↑
            选哪个??

目标：
- 均匀分配请求（不让某台机器过载）
- 同一请求尽量路由到同一节点（有状态场景）
```

### 1.2 本项目的负载均衡体系

```java
// LoadBalance.java — 顶层接口
public interface LoadBalance {
    /**
     * 从服务地址列表中选出一个
     * @param serviceAddresses 所有可用服务地址
     * @param rpcServiceName   服务名（一致性哈希需要）
     * @return 选中的地址
     */
    String selectServiceAddress(List<String> serviceAddresses, String rpcServiceName);
}
```

### 1.3 随机负载均衡（RandomLoadBalance）

```java
public class RandomLoadBalance extends AbstractLoadBalance {
    @Override
    protected String doSelect(List<String> serviceAddresses, String rpcServiceName) {
        Random random = new Random();
        return serviceAddresses.get(random.nextInt(serviceAddresses.size()));
    }
}
```

- 优点：简单，不需要维护状态
- 缺点：完全随机，不关心节点负载
- 适用于：无状态、节点性能相同的场景

### 1.4 一致性哈希负载均衡（ConsistentHashLoadBalance）★重点

这是面试中最常被深挖的算法。

**问题场景**：如果用户 A 的会话数据在节点 1，下次请求路由到节点 2，会话就丢了。

**一致性哈希的解决方案**：

```
将整个哈希空间组织成一个虚拟的环（0 ~ 2^32-1）：

              0
         ┌─────────┐
        /           \
       /   Node1     \
      /   (hash=100)  \
     │                 │
     │     Node3       │
     │   (hash=3000)   │
     │                 │
      \    Node2      /
       \  (hash=6000) /
        \            /
         └──────────┘
            2^32-1

请求过来 → 计算 hash(key) → 顺时针找到第一个节点 → 路由过去
Node2 宕机 → 原本到 Node2 的请求顺移到 Node1 → 只影响少部分请求
```

```java
public class ConsistentHashLoadBalance extends AbstractLoadBalance {
    // ★ 虚拟节点：每个真实节点映射为多个虚拟节点，使分布更均匀
    private static final int VIRTUAL_NODE_NUM = 3;

    @Override
    protected String doSelect(List<String> serviceAddresses, String serviceName) {
        // ★ 每次选择时重建哈希环（因为节点列表可能变了）
        // 生产环境可以缓存并配合 Watch 机制重建
        TreeMap<Integer, String> ring = new TreeMap<>();

        for (String address : serviceAddresses) {
            for (int i = 0; i < VIRTUAL_NODE_NUM; i++) {
                int hash = hash(address + "#" + i);  // ★ 虚拟节点
                ring.put(hash, address);
            }
        }

        // 计算请求的哈希值
        int requestHash = hash(serviceName);
        // 顺时针找第一个节点（大于等于当前 hash 的第一个节点）
        Map.Entry<Integer, String> entry = ring.ceilingEntry(requestHash);
        if (entry == null) {
            // 超过了最大值 → 回到环的起点
            entry = ring.firstEntry();
        }
        return entry.getValue();
    }
}
```

**关键概念**：

| 概念 | 说明 | 面试用语 |
|------|------|----------|
| **哈希环** | 0 ~ 2^32-1 的圆形空间 | 「将节点和请求映射到同一个哈希空间」 |
| **虚拟节点** | 一个真实节点对应多个虚拟节点 | 「解决数据倾斜问题，使分布更均匀」 |
| **顺时针查找** | TreeMap.ceilingEntry() | 「时间复杂度 O(log N)」 |
| **影响范围** | 增删节点只影响相邻节点 | 「避免全局重新分配」 |

## 二、心跳检测

### 2.1 为什么需要心跳

TCP 连接可能「假死」：
- 网络闪断，TCP 没有及时感知
- 对端进程卡死，但仍然能响应 TCP keep-alive
- 防火墙静默断开空闲连接

### 2.2 本项目的心跳实现

```java
// NettyRpcServer.java 启动时配置
ch.pipeline().addLast(
    new IdleStateHandler(
        30,    // ★ 读空闲 30 秒 → 触发事件
        0,     // 写空闲不检测
        0,     // 读写空闲不检测
        TimeUnit.SECONDS
    )
);
```

触发流程：
```
1. Server 端 30 秒没收到数据 → IdleStateHandler 触发 READER_IDLE 事件
2. NettyRpcServerHandler.userEventTriggered() 捕捉到事件
3. 判定：这个连接可能已断 → 关闭 Channel
4. Client 端 ChannelInactive → 创建新连接
```

**面试问**：「30 秒怎么定的？」
> 「权衡了故障发现速度和误判率。太短（如 5 秒）可能因网络抖动误判断开，太长（如 5 分钟）则故障感知太慢。30 秒是常见的折中选择。」

## 三、Gzip 压缩

### 3.1 压缩的意义

RPC 调用中，序列化后的数据可能很大（如传输大对象列表），压缩可以显著减少网络传输量。

### 3.2 本项目实现

```java
// Compress.java — 顶层接口
public interface Compress {
    byte[] compress(byte[] bytes);
    byte[] decompress(byte[] bytes);
}

// GzipCompress.java
public class GzipCompress implements Compress {
    private static final int BUFFER_SIZE = 1024 * 4;  // 4KB 缓冲区

    @Override
    public byte[] compress(byte[] bytes) {
        ByteArrayOutputStream out = new ByteArrayOutputStream();
        try (GZIPOutputStream gzip = new GZIPOutputStream(out)) {
            gzip.write(bytes);
            gzip.flush();
            gzip.finish();
            return out.toByteArray();
        }
    }

    @Override
    public byte[] decompress(byte[] bytes) {
        ByteArrayOutputStream out = new ByteArrayOutputStream();
        try (GZIPInputStream gzip = new GZIPInputStream(new ByteArrayInputStream(bytes))) {
            byte[] buffer = new byte[BUFFER_SIZE];
            int n;
            while ((n = gzip.read(buffer)) > -1) {
                out.write(buffer, 0, n);
            }
        }
        return out.toByteArray();
    }
}
```

**压缩收益**：
- 文本内容（如 JSON 序列化后的数据）：压缩比可达 80%
- 二进制数据（如图片）：压缩比通常不高，约 10-20%
- 已经用 Kryo 做了高效序列化，再 Gzip 压缩可能收益有限 → **用之前需要评估是否值得**

## 四、服务分组与版本控制

### 4.1 服务分组（Group）

场景：同一个接口 `HelloService`，有开发环境和测试环境两个实现。

```java
// 开发环境实现
@RpcService(group = "dev", version = "1.0")
public class DevHelloServiceImpl implements HelloService { ... }

// 测试环境实现
@RpcService(group = "test", version = "1.0")
public class TestHelloServiceImpl implements HelloService { ... }

// 调用方指定使用哪个分组
@RpcReference(group = "test", version = "1.0")
private HelloService helloService;
```

### 4.2 服务版本（Version）

场景：接口升级，新旧版本需要同时在线。

```
RpcRequest 中携带 group + version
    ↓
注册路径包含 /group/version
    ↓
服务发现时根据 group + version 过滤
    ↓
调用方精确匹配到对应版本的服务
```

## 五、SPI 扩展加载器详解

### 5.1 ExtensionLoader 工作流程

```java
public class ExtensionLoader<T> {
    private static final String EXTENSION_DIR = "META-INF/extensions/";

    // ★ 核心方法：根据接口类型和实现名称获取实例
    public T getExtension(String name) {
        // 1. 读取配置文件
        Map<String, Class<?>> extensionClasses = getExtensionClasses();

        // 2. 根据 name 找到对应实现类
        Class<?> clazz = extensionClasses.get(name);

        // 3. 反射创建实例并缓存（单例）
        Holder<Object> holder = cachedInstances.get(name);
        if (holder == null) {
            holder = new Holder<>();
            holder.set(clazz.getDeclaredConstructor().newInstance());
            cachedInstances.put(name, holder);
        }
        return (T) holder.get();
    }

    // 读取 META-INF/extensions/接口名 文件
    private Map<String, Class<?>> getExtensionClasses() {
        String fileName = EXTENSION_DIR + type.getName();
        // 使用 ClassLoader 读取配置文件
        Enumeration<URL> urls = ClassLoader.getSystemResources(fileName);
        // 解析每一行的 key=value 格式
        while (urls.hasMoreElements()) {
            // key = "kryo", value = "com.zhang.serialize.kyro.KryoSerializer"
        }
    }
}
```

**配置文件示例**：
```
# META-INF/extensions/com.zhang.serialize.Serializer
kryo=com.zhang.serialize.kyro.KryoSerializer
protostuff=com.zhang.serialize.protostuff.ProtostuffSerializer
hessian=com.zhang.serialize.hessian.HessianSerializer
```

### 5.2 对标 Dubbo 的 SPI

Dubbo 的 SPI 思想与本项目一致，但更强：
- Dubbo 的 SPI 支持**依赖注入**和 **AOP 包装**
- Dubbo 的 SPI 支持**自适应扩展**（@Adaptive 注解）

在面试时可以这样说：
> 「我实现的 SPI 机制是对 Dubbo SPI 思想的简化版，虽然只支持基础的服务发现，但原理和 Dubbo 是一致的：通过配置文件 + 反射实现运行时动态加载实现类。」

## 六、本章面试要点

1. **一致性哈希的原理和优势？** → 哈希环 + 顺时针查找，节点增减只影响相邻节点
2. **虚拟节点解决什么问题？** → 当节点数少时哈希环分布不均，虚拟节点让分布更均匀
3. **心跳检测怎么实现的？目的？** → IdleStateHandler 检测读写空闲，防止连接假死
4. **压缩在 RPC 中的作用？** → 减少网络传输量，但需权衡压缩/解压缩的 CPU 开销
5. **Group 和 Version 的应用场景？** → Group 区分环境/多实现，Version 管理版本兼容
6. **SPI 机制的价值？** → 运行时动态加载，新增实现不需要改核心代码，符合开闭原则
