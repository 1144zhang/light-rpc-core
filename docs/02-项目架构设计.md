# 项目架构设计

## 一、项目概览

`light-rpc-core` 是一款基于 **Netty + Zookeeper + Kryo** 的轻量级 RPC 框架，采用多模块 Maven 工程结构，模块职责清晰，分层明确。

### 核心设计目标

| 目标 | 说明 |
|------|------|
| **透明化** | 调用方感知不到远程和本地的区别（动态代理） |
| **高性能** | Netty NIO + Kryo 二进制序列化 + 连接复用 |
| **可扩展** | SPI 机制实现序列化方式可插拔 |
| **高可用** | ZK 注册中心 + 负载均衡 + 心跳检测 |
| **易集成** | Spring 注解驱动，零侵入接入业务代码 |

## 四、核心数据流

### 4.1 请求流（Client → Server）

```
用户调用 helloService.hello(msg)
        │
        ▼
RpcClientProxy.invoke()          ← JDK 动态代理拦截
        │
        ▼
从 ZK 获取服务地址列表
        │
        ▼
负载均衡选出一个节点              ← ConsistentHashLoadBalance
        │
        ▼
ChannelProvider 获取/创建 Channel ← 连接复用
        │
        ▼
构建 RpcRequest                  ← interfaceName + methodName + params + requestId
        │
        ▼
RpcMessageEncoder 编码           ← 自定义协议封装 + 序列化
        │
        ▼
Netty 发送 (writeAndFlush)       ← TCP 传输
        │
        ▼
CompletableFuture 挂起等待       ← requestId → future 映射
```

### 4.2 响应流（Server → Client）

```
Netty Server 收到字节流
        │
        ▼
RpcMessageDecoder 解码           ← 自定义协议解析 + 反序列化
        │
        ▼
RpcRequestHandler 处理           ← 反射调用目标方法
        │
        ▼
构建 RpcResponse                 ← requestId + code + data
        │
        ▼
RpcMessageEncoder 编码返回       ← 序列化 + 协议封装
        │
        ▼
Client 收到响应                   ← NettyRpcClientHandler
        │
        ▼
CompletableFuture.complete()     ← 根据 requestId 匹配，唤醒等待
        │
        ▼
返回给调用方
```

## 五、关键设计决策

### 5.1 为什么用 Netty（NIO）而不是 Socket（BIO）

| 对比 | BIO (Socket) | NIO (Netty) |
|------|-------------|------------|
| 线程模型 | 一个连接一个线程 | Reactor 模型，少量线程 |
| 连接数 | 受线程数限制 | 可支撑万级连接 |
| 吞吐量 | 低（线程上下文切换） | 高（事件驱动） |
| 零拷贝 | 不支持 | 支持（sendfile） |

本项目在迭代过程中，**最初用 Socket 实现**，然后优化为 Netty，正是为了展示这个演进过程的理解。

### 5.2 为什么自定义协议而不是直接用 JSON

```
自定义协议格式：
┌──────────┬──────────┬──────────┬──────────┐
│  Magic   │Version+  │ Body     │  Body    │
│  4 Bytes │Serializer│ Length   │  (bytes) │
│          │ 1 Byte   │ 4 Bytes  │          │
└──────────┴──────────┴──────────┴──────────┘
```

- **魔数（Magic Number）**：快速识别是否为有效 RPC 消息
- **版本 + 序列化器编号**：1 字节搞定版本协商和序列化方式
- **消息体长度**：解决 TCP 粘包/拆包问题
- **总头部仅 9 字节**，而 HTTP 头部动辄几百字节

### 5.3 为什么用 Zookeeper 做注册中心

- **CP 模型**（一致性 + 分区容错性）：保证服务列表的强一致性
- **临时节点**：服务端断连后自动清除，天然支持服务健康检查
- **Watch 机制**：服务列表变更时主动通知客户端，无需轮询

### 5.4 为什么用 CompletableFuture 而不是 Object.wait()

旧实现：`AttributeMap` 绑定到 Channel 上，请求完成后手动 notify → **线程阻塞，实现复杂**

新实现：`UnprocessedRequests` 维护 `Map<String, CompletableFuture<RpcResponse>>`

```java
// 发送请求时
CompletableFuture<RpcResponse> future = new CompletableFuture<>();
UNPROCESSED_REQUESTS.put(requestId, future);
channel.writeAndFlush(rpcRequest);

// 收到响应时
RpcResponse response = ...;
UNPROCESSED_REQUESTS.complete(requestId, response);
```

优势：
- 非阻塞，释放 I/O 线程
- 支持链式调用 `.thenApply()` `.thenAccept()`
- 支持超时 `.get(5, TimeUnit.SECONDS)`

## 六、设计模式应用

本项目运用了多种设计模式，面试时能讲出来很加分：

| 设计模式 | 应用位置 | 说明 |
|----------|----------|------|
| **代理模式** | `RpcClientProxy` | JDK 动态代理，屏蔽远程调用细节 |
| **单例模式** | `SingletonFactory` | 双重校验锁，保证全局唯一实例 |
| **工厂模式** | `ThreadPoolFactoryUtil` | 统一创建线程池 |
| **模板方法** | `AbstractLoadBalance` | 定义算法骨架，子类实现具体选择逻辑 |
| **策略模式** | `LoadBalance` 接口 | 不同负载均衡算法可自由切换 |
| **装饰者模式** | `Compress` 接口 | Gzip 压缩作为装饰层 |
| **SPI（服务发现）** | `ExtensionLoader` | 类似 Dubbo 的可插拔扩展机制 |

## 七、本章面试要点

1. **项目架构是什么样的？** → 分层设计：代理层 → 治理层 → 传输层 → 公共层
2. **模块怎么划分的？为什么这样分？** → common（公共依赖）+ simple（核心实现）+ example（示例）
3. **数据流是怎样的？** → 代理拦截 → 发现服务 → 负载均衡 → 序列化 → 编码 → Netty 发送 → 解码 → 反射调用 → 返回
4. **为什么自定义协议？** → 比 HTTP 更紧凑（9 字节头部 vs 几百字节），包含魔数安全校验，长度字段解决粘包
5. **用了哪些设计模式？** → 代理、单例、工厂、模板方法、策略、装饰者、SPI
