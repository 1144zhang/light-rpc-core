# 序列化机制详解

## 一、什么是序列化？为什么需要？

**序列化**：把内存中的对象变成可以在网络上传输的字节流。
**反序列化**：把字节流还原成内存中的对象。

```java
对象 → 序列化 → 字节流 → 网络传输 → 反序列化 → 对象
```

### 为什么 JSON 不够用

面试官问「为什么不用 JSON？」时，可以这样回答：

| 对比维度 | JSON | Kryo（二进制） |
|----------|------|---------------|
| 体积 | 大（字段名重复存储） | 小（字段名不存储，用序号代替） |
| 速度 | 慢（字符串解析） | 快（直接读写二进制） |
| 类型安全 | 弱（所有数字都是 number） | 强（精确类型匹配） |
| 跨语言 | 天然支持 | 需额外适配 |
| 可读性 | 人可读 | 不可读 |

示例对比（同一个对象）：

```json
// JSON 格式：~150 字节
{"interfaceName":"com.zhang.HelloService","methodName":"hello",
"parameters":[{"message":"你好","description":"测试消息"}],
"paramTypes":["com.zhang.Hello"],"requestId":"abc-123"}
```

```
// Kryo 二进制：~40 字节（减少 70%+）
[0x01][0x17]com.zhang.HelloService[0x05]hello[0x01][0x06]你好[0x0c]测试消息
```

## 二、本项目的序列化体系

### 2.1 整体设计

```
                    ┌───────────────┐
                    │   Serializer  │  ← 顶层接口
                    │   (接口)       │
                    └───────┬───────┘
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
┌───────▼───────┐  ┌───────▼───────┐  ┌───────▼───────┐
│ KryoSerializer│  │ProtostuffSer  │  │HessianSerializer│
│               │  │               │  │                  │
│ 高性能二进制   │  │ Google 出品    │  │ Caucho 出品      │
│ 需ThreadLocal  │  │ 无需注册类     │  │ 兼容性好         │
└───────────────┘  └───────────────┘  └──────────────────┘
```

### 2.2 Serializer 接口

```java
// Serializer.java
public interface Serializer {
    // 序列化：对象 → 字节数组
    byte[] serialize(Object obj);

    // 反序列化：字节数组 → 对象
    <T> T deserialize(byte[] bytes, Class<T> clazz);
}
```

设计要点：
- 极简接口，只有 2 个方法
- 序列化和反序列化都返回/接收字节数组，与传输层解耦
- 通过泛型 `<T>` 保证类型安全

## 三、KryoSerializer — 核心实现

### 3.1 为什么选 Kryo

- **高性能**：直接操作字节码生成序列化逻辑，比反射快
- **小体积**：不存字段名，用序号索引
- **成熟稳定**：Twitter、Apache Storm 等大量使用

### 3.2 关键问题：Kryo 不是线程安全的

```
错误示例（多线程共享一个 Kryo 实例）：
┌──────────┐  ┌──────────┐  ┌──────────┐
│ Thread A  │  │ Thread B  │  │ Thread C  │
│ serialize │  │ serialize │  │ serialize │
└─────┬─────┘  └─────┬─────┘  └─────┬─────┘
      │               │               │
      └───────────────┼───────────────┘
                      │
              ┌───────▼───────┐
              │   同一个 Kryo  │  ← 内部状态被并发修改
              │   实例（不安全） │     导致数据错乱!
              └───────────────┘
```

### 3.3 解决方案：ThreadLocal 隔离

```java
// KryoSerializer.java — 核心设计
public class KryoSerializer implements Serializer {
    // ★ 每个线程持有一个独立的 Kryo 实例
    private static final ThreadLocal<Kryo> KRYO_THREAD_LOCAL = 
        ThreadLocal.withInitial(() -> {
            Kryo kryo = new Kryo();
            kryo.setRegistrationRequired(false);  // 不需要预注册所有类
            return kryo;
        });

    @Override
    public byte[] serialize(Object obj) {
        ByteArrayOutputStream bos = new ByteArrayOutputStream();
        Output output = new Output(bos);
        Kryo kryo = KRYO_THREAD_LOCAL.get();  // 获取当前线程的 Kryo
        kryo.writeObject(output, obj);
        output.close();
        return bos.toByteArray();
    }

    @Override
    public <T> T deserialize(byte[] bytes, Class<T> clazz) {
        ByteArrayInputStream bis = new ByteArrayInputStream(bytes);
        Input input = new Input(bis);
        Kryo kryo = KRYO_THREAD_LOCAL.get();  // 获取当前线程的 Kryo
        T obj = kryo.readObject(input, clazz);
        input.close();
        return obj;
    }
}
```

**设计要点**：
- 每个线程有自己独立的 Kryo 实例（ThreadLocal）
- Kryo 的创建有开销，ThreadLocal 保证创建一次、复用多次
- 用 `withInitial()` 延迟初始化

## 四、其他序列化实现对比

### 4.1 ProtostuffSerializer

Google 出品，基于 Protobuf 但更易用：

```java
// ProtostuffSerializer.java
public byte[] serialize(Object obj) {
    LinkedBuffer buffer = LinkedBuffer.allocate(LinkedBuffer.DEFAULT_BUFFER_SIZE);
    try {
        Schema schema = RuntimeSchema.getSchema(obj.getClass());  // 运行时获取 Schema
        return ProtostuffIOUtil.toByteArray(obj, schema, buffer);
    } finally {
        buffer.clear();
    }
}
```

特点：无需 `.proto` 文件，运行时自动推断 Schema

### 4.2 HessianSerializer

Caucho 出品，Dubbo 默认序列化方式：

```java
// HessianSerializer.java
public byte[] serialize(Object obj) {
    ByteArrayOutputStream bos = new ByteArrayOutputStream();
    Hessian2Output out = new Hessian2Output(bos);
    out.writeObject(obj);
    out.close();
    return bos.toByteArray();
}
```

特点：兼容性好，支持 Java 序列化接口

### 4.3 性能对比（面试时可以背的数据）

| 序列化方式 | 耗时 | 体积 | 是否跨语言 |
|-----------|------|------|-----------|
| JDK 原生 | 100% (基准) | 100% (基准) | ❌ |
| JSON (Jackson) | ~60% | ~80% | ✅ |
| Hessian2 | ~40% | ~40% | 部分 |
| Protostuff | ~15% | ~20% | ✅ (需 schema) |
| Kryo | ~10% | ~15% | ❌ |

## 五、SPI 扩展机制在序列化中的应用

### 5.1 什么是 SPI

SPI（Service Provider Interface）：一种**服务发现机制**，通过在 classpath 下配置实现类，运行时动态加载。

### 5.2 本项目的 SPI 实现

**第 1 步：定义接口和注解**

```java
// Serializer.java（接口）
public interface Serializer { ... }

// SPI.java（标记注解，用于标记"这是一个 SPI 接口"）
@Documented
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface SPI { }
```

**第 2 步：配置文件**

```
# rpc-framework-simple/src/main/resources/META-INF/extensions/com.zhang.serialize.Serializer

kryo=com.zhang.serialize.kyro.KryoSerializer
protostuff=com.zhang.serialize.protostuff.ProtostuffSerializer
hessian=com.zhang.serialize.hessian.HessianSerializer
```

**第 3 步：运行时加载**

```java
// ExtensionLoader.java — 核心逻辑
public class ExtensionLoader<T> {
    private static final String EXTENSION_DIR = "META-INF/extensions/";

    public T getExtension(String name) {
        // 1. 根据接口全限定名，找到对应的配置文件
        String fileName = EXTENSION_DIR + interfaceClass.getName();
        // 2. 读取配置文件，解析 key=value 映射
        // 3. 反射实例化对应的实现类
        Class<?> clazz = Class.forName(className);
        return (T) clazz.getDeclaredConstructor().newInstance();
    }
}
```

### 5.3 这样做的好处

- **解耦**：新增序列化方式只需加一个实现类 + 一行配置，不用改任何核心代码
- **可配置**：用户可以通过配置文件选择序列化方式
- **对标 Dubbo**：与 Dubbo 的 SPI 扩展设计思想一致

## 六、本章面试要点

1. **为什么不用 JSON 做序列化？** → 体积大、速度慢、缺乏类型安全，JSON 适合对外 API，RPC 追求高性能用二进制
2. **Kryo 比 JDK 序列化快在哪？** → JDK 序列化需要反射获取所有字段元数据，Kryo 直接生成字节码；Kryo 不存类结构信息，体积小很多
3. **Kryo 线程不安全怎么解决？** → ThreadLocal 为每个线程创建独立实例
4. **你对比过哪些序列化方式？** → 见表，Kryo 最快，Protostuff 跨语言最强
5. **SPI 机制是什么？有什么好处？** → 服务发现机制，运行时动态加载实现类，解耦 + 可扩展
6. **为什么序列化接口只设计 2 个方法？** → 接口最小化原则，职责单一，只做对象 ↔ 字节转换
