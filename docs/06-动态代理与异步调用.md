# 动态代理与异步调用

## 一、RPC 的核心难题：如何让远程调用「透明」

### 1.1 没有代理时

```java
// 调用方需要手动完成所有步骤：
RpcRequest request = new RpcRequest("HelloService", "hello", params);
byte[] bytes = serializer.serialize(request);
channel.writeAndFlush(bytes);
// ... 等待响应、反序列化、提取结果
```

这种方式让调用方直接接触了网络通信、序列化等底层细节，业务代码和 RPC 逻辑高度耦合。

### 1.2 理想状态

```java
// 调用方只关心业务：
@RpcReference
private HelloService helloService;  // 像本地 Bean 一样注入

String result = helloService.hello(new Hello("张三"));  // 像本地方法一样调用
```

底层发生了什么？**动态代理**。

## 二、JDK 动态代理原理

### 2.1 代理模式

```
调用方                代理对象              真实对象
  │                     │                    │
  │  hello("张三")      │                    │
  │────────────────────>│                    │
  │                     │  1. 构建 RpcRequest│
  │                     │  2. 序列化          │
  │                     │  3. 网络发送        │
  │                     │  4. 等待响应        │
  │                     │  5. 反序列化        │
  │  返回 "你好 张三"    │                    │
  │<────────────────────│                    │
  │                     │                    │
 感知不到远程调用的复杂性  所有脏活累活在这里       │
```

### 2.2 JDK 动态代理三要素

```java
// Proxy.newProxyInstance(ClassLoader, 接口列表, InvocationHandler)

Proxy.newProxyInstance(
    clazz.getClassLoader(),           // 1. 类加载器
    new Class<?>[]{clazz},            // 2. 被代理的接口
    new InvocationHandler() {         // 3. 调用处理器：每次调用都会被拦截到这里
        @Override
        public Object invoke(Object proxy, Method method, Object[] args) {
            // ★ 在这里完成 RPC 远程调用的全部逻辑
        }
    }
);
```

## 三、本项目的核心实现：RpcClientProxy

```java
// RpcClientProxy.java — 核心类
public class RpcClientProxy implements InvocationHandler {

    private final ServiceDiscovery serviceDiscovery;
    private final RpcRequestTransport rpcRequestTransport;

    /**
     * 获取代理对象
     * 调用方通过这个方法拿到一个「伪本地对象」
     */
    @SuppressWarnings("unchecked")
    public <T> T getProxy(Class<T> clazz) {
        return (T) Proxy.newProxyInstance(
            clazz.getClassLoader(),
            new Class<?>[]{clazz},
            this     // ★ 每个代理对象复用同一个 InvocationHandler
        );
    }

    /**
     * ★ 核心拦截逻辑
     * 调用方的每一次方法调用，都会被拦截到这里
     */
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) {
        // 1. 构建 RpcRequest
        RpcRequest rpcRequest = RpcRequest.builder()
            .interfaceName(method.getDeclaringClass().getName())  // 接口名
            .methodName(method.getName())                         // 方法名
            .parameters(args)                                     // 参数值
            .paramTypes(method.getParameterTypes())               // 参数类型
            .requestId(UUID.randomUUID().toString())              // ★ 唯一请求ID
            .group(group)
            .version(version)
            .build();

        // 2. 通过注册中心发现服务
        List<String> serviceAddresses = 
            serviceDiscovery.lookupService(rpcRequest.getServiceName());

        // 3. 负载均衡选出一个提供者
        String targetAddress = loadBalance.selectServiceAddress(serviceAddresses);

        // 4. 通过网络发送请求，返回 CompletableFuture
        CompletableFuture<RpcResponse<Object>> future = 
            (CompletableFuture<RpcResponse<Object>>) rpcRequestTransport.sendRpcRequest(rpcRequest);

        // 5. 阻塞等待结果（也可以改为异步）
        RpcResponse<Object> response = future.get();

        // 6. 返回结果给调用方
        return response.getData();
    }
}
```

## 四、异步调用机制：CompletableFuture

### 4.1 问题：如何把「异步网络 I/O」和「同步调用语义」结合起来

```
时间线：

Client 线程                         Netty I/O 线程
    │                                    │
    │ 1. sendRpcRequest()                │
    │    创建 CompletableFuture           │
    │    存入 Map<requestId, future>      │
    │    channel.writeAndFlush()         │
    │                                    │
    │ 2. future.get()   ← 阻塞等待        │
    │       │                            │
    │       │                    3. 收到 Response
    │       │                    从 Map 取出 future
    │       │                    future.complete(response)
    │       │                            │
    │ 4. 被唤醒，拿到结果  ←───────────────┘
    │    返回给调用方
```

### 4.2 核心实现：UnprocessedRequests

```java
// UnprocessedRequests.java
public class UnprocessedRequests {
    // ★ requestId → CompletableFuture 的映射
    private static final Map<String, CompletableFuture<RpcResponse<Object>>> 
        UNPROCESSED_RESPONSE_FUTURES = new ConcurrentHashMap<>();

    // 发送请求时调用：将 future 放入 Map
    public void put(String requestId, CompletableFuture<RpcResponse<Object>> future) {
        UNPROCESSED_RESPONSE_FUTURES.put(requestId, future);
    }

    // 收到响应时调用：从 Map 取出 future 并完成
    public void complete(RpcResponse<Object> rpcResponse) {
        CompletableFuture<RpcResponse<Object>> future = 
            UNPROCESSED_RESPONSE_FUTURES.remove(rpcResponse.getRequestId());
        if (future != null) {
            future.complete(rpcResponse);  // ★ 唤醒阻塞在 future.get() 的线程
        }
    }
}
```

### 4.3 为什么用 ConcurrentHashMap

- **多线程安全**：多个 Client 线程同时发送请求，并发 put/find
- **高性能**：分段锁，并发度高于 Hashtable
- **remove 原子性**：取出 future 的同时从 Map 中删除，防止内存泄漏

### 4.4 NettyRpcClient 中的发送逻辑

```java
// NettyRpcClient.java — sendRpcRequest 方法
public Object sendRpcRequest(RpcRequest rpcRequest) {
    // 1. 创建 CompletableFuture
    CompletableFuture<RpcResponse<Object>> resultFuture = new CompletableFuture<>();

    // 2. 获取服务地址（负载均衡）
    InetSocketAddress address = serviceDiscovery.lookupService(rpcRequest);

    // 3. 获取或创建 Channel
    Channel channel = channelProvider.get(address);

    if (channel.isActive()) {
        // 4. 存入未完成请求 Map
        unprocessedRequests.put(rpcRequest.getRequestId(), resultFuture);

        // 5. 构建 RpcMessage 并发送
        RpcMessage rpcMessage = RpcMessage.builder()
            .data(rpcRequest)
            .codec(SerializerTypeEnum.KRYO.getCode())
            .build();

        channel.writeAndFlush(rpcMessage).addListener((ChannelFutureListener) future -> {
            if (!future.isSuccess()) {
                // 发送失败 → 也要清理 future，否则调用方无限等待
                unprocessedRequests.complete(
                    RpcResponse.fail(rpcRequest.getRequestId(), future.cause().getMessage())
                );
            }
        });
    }

    return resultFuture;  // ★ 返回 future，调用方通过 future.get() 等待结果
}
```

## 五、异步 vs 同步调用方式

本项目目前支持**默认异步（CompletableFuture）**，调用方可选择同步或异步：

```java
// 同步调用（默认）
RpcResponse<Object> response = rpcRequestTransport.sendRpcRequest(request).get();
return response.getData();

// 异步调用（可选）
CompletableFuture<RpcResponse<Object>> future = rpcRequestTransport.sendRpcRequest(request);
future.thenAccept(response -> {
    // 收到响应的回调处理
    System.out.println("Result: " + response.getData());
});
// 主线程不阻塞，继续做其他事
```

## 六、本章面试要点

1. **动态代理在 RPC 中起什么作用？** → 屏蔽远程调用细节，让调用方像使用本地对象一样使用远程服务
2. **JDK 动态代理的原理是什么？** → 运行时生成代理类字节码，继承 Proxy 类，实现目标接口，拦截所有方法调用
3. **JDK 代理和 CGLIB 代理的区别？** → JDK 基于接口，CGLIB 基于继承（可代理普通类）；JDK 创建快运行稍慢，CGLIB 创建慢运行快
4. **requestId 的作用是什么？** → 关联请求和响应，因为一个 Channel 上可能同时发出多个请求
5. **CompletableFuture 怎么实现请求-响应匹配？** → requestId 作为 key 存入 ConcurrentHashMap，收到响应时根据 requestId 找到对应的 future 并 complete
6. **如果服务端没有返回结果，Client 会怎样？** → future.get() 会一直阻塞，需要加超时机制（`.get(5, TimeUnit.SECONDS)`）
7. **ConcurrentHashMap 为什么适合这个场景？** → 线程安全 + 高性能 + remove 原子性
