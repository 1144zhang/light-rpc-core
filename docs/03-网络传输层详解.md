# 网络传输层详解

## 一、演进路线：从 Socket (BIO) 到 Netty (NIO)

本项目的网络层经历了**两次迭代**，这也是面试时最能展示技术深度的部分：

```
V1.0: Java Socket（BIO）             → 简单但不能支撑高并发
V2.0: Netty（NIO + Reactor 模型）    → 高性能、可扩展
```

### 1.1 V1.0 — Socket BIO 版本

```java
// SocketRpcServer.java（原始版本）
ServerSocket serverSocket = new ServerSocket(port);
while (true) {
    Socket socket = serverSocket.accept();  // 阻塞等待连接
    // 每来一个连接，起一个新线程
    new Thread(new SocketRpcRequestHandlerRunnable(socket)).start();
}
```

**问题**：
- `accept()` 阻塞：没有连接时线程一直等
- 一个连接一个线程：1000 个连接 = 1000 个线程 = 大量上下文切换
- `InputStream.read()` 阻塞：等待数据时线程挂起

### 1.2 V2.0 — Netty NIO 版本

```java
// NettyRpcServer.java（优化版本）
ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap.group(bossGroup, workerGroup)     // 主从 Reactor 线程池
         .channel(NioServerSocketChannel.class)  // NIO
         .childHandler(new ChannelInitializer<SocketChannel>() {
             @Override
             protected void initChannel(SocketChannel ch) {
                 ch.pipeline()
                   .addLast(new RpcMessageFrameDecoder())   // 帧解码
                   .addLast(new RpcMessageDecoder())         // 消息解码
                   .addLast(new RpcMessageEncoder())         // 消息编码
                   .addLast(new NettyRpcServerHandler());   // 业务处理
             }
         });
```

**改进**：
- Reactor 模型：少量线程处理大量连接
- 事件驱动：有数据才处理，不阻塞
- ChannelPipeline：责任链模式，编解码与业务逻辑解耦

## 二、Netty 核心概念

### 2.1 Reactor 线程模型

```
                    ┌─────────────┐
                    │  Boss Group  │  只负责接受连接（通常 1 个线程）
                    │  (NioEventLoop)│
                    └──────┬──────┘
                           │ 收到新连接后注册到 Worker
          ┌────────────────┼────────────────┐
          │                │                │
    ┌─────▼─────┐   ┌─────▼─────┐   ┌─────▼─────┐
    │  Worker 1 │   │  Worker 2 │   │  Worker N │  负责读写处理
    │  EventLoop│   │  EventLoop│   │  EventLoop│  (默认 CPU 核数×2)
    └───────────┘   └───────────┘   └───────────┘
```

- **Boss Group**：只负责 `accept` 新连接，然后注册到 Worker
- **Worker Group**：负责已建立连接的 I/O 读写和业务处理
- 每个 EventLoop 绑定一个线程，负责多个 Channel，**避免了线程切换开销**

### 2.2 ChannelPipeline 责任链

```
┌──────────────────────────────────────────────┐
│            ChannelPipeline                    │
│                                              │
│  Inbound（入站，接收数据）：                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │Frame     │→│ Decoder  │→│ Handler  │    │
│  │Decoder   │  │          │  │          │    │
│  └──────────┘  └──────────┘  └──────────┘    │
│      拆帧         反序列化      业务处理          │
│                                              │
│  Outbound（出站，发送数据）：                      │
│  ┌──────────┐  ┌──────────┐                  │
│  │ Encoder  │←│ Handler  │                  │
│  │          │  │  (write) │                  │
│  └──────────┘  └──────────┘                  │
│      序列化       业务发起                       │
└──────────────────────────────────────────────┘
```

本项目的 Pipeline 顺序（关键！顺序错了一切都乱）：

```java
ch.pipeline()
  // ★ 必须最先：帧解码器，解决粘包拆包
  .addLast(new RpcMessageFrameDecoder())
  // ★ 然后：自定义协议解码器（将字节转为 RpcMessage 对象）
  .addLast(new RpcMessageDecoder())
  // ★ 编码器（出站时调用，将 RpcMessage 对象转为字节）
  .addLast(new RpcMessageEncoder())
  // ★ 最后：业务处理器
  .addLast(new NettyRpcServerHandler());
```

## 三、自定义通信协议

### 3.1 协议格式设计

```
  0        4        5        9        13          ...
┌────────┬────────┬────────┬────────┬─────────────┐
│  Magic │Version+│  Body  │  Body  │    Body     │
│ Number │SerType │ Length │  Data  │    Data     │
│ 4Bytes │ 1Byte  │ 4Bytes │  N Bytes (变长)      │
└────────┴────────┴────────┴────────┴─────────────┘

总头部：9 字节（固定）
Body 长度：由 Length 字段指定
```

### 3.2 各字段含义

| 字段 | 字节 | 说明 | 为什么要 |
|------|------|------|----------|
| **Magic Number** | 4 | 魔数，固定值 `0xCAFEBABE` | 快速识别无效数据包，过滤掉非法请求 |
| **Version + Serializer** | 1 | 高 4 位 = 版本号，低 4 位 = 序列化器编号 | 1 字节搞定版本协商和序列化方式选择 |
| **Body Length** | 4 | 消息体字节数 | **解决 TCP 粘包/拆包的关键** |
| **Body Data** | N | Protobuf/Kryo 序列化后的 RpcRequest/RpcResponse | 实际传输的数据 |

### 3.3 对应代码

本项目的协议编解码逻辑封装在 `RpcMessageCodec.java` 中：

```java
// RpcMessageCodec.java — 编码方法
protected void encode(ChannelHandlerContext ctx, RpcMessage msg, ByteBuf out) {
    // 1. 4 字节魔数
    out.writeBytes(RpcConstants.MAGIC_NUMBER);           // [0xcafebabe]
    // 2. 1 字节：版本(高4位) + 序列化器编号(低4位)
    out.writeByte(msg.getCodec());                        // [0x01|0x00]
    // 3. 4 字节消息体长度
    byte[] bodyBytes = ...;  // 序列化后的字节
    out.writeInt(bodyBytes.length);                       // [body length]
    // 4. 消息体
    out.writeBytes(bodyBytes);                            // [data]
}

// RpcMessageCodec.java — 解码方法
protected RpcMessage decode(ChannelHandlerContext ctx, ByteBuf in) {
    // 0. 检查可读字节数是否 ≥ 消息头部长度（9 字节）
    if (in.readableBytes() < RpcConstants.HEAD_LENGTH) return null;

    // 1. 读取并校验魔数
    int magic = in.readInt();
    if (magic != RpcConstants.MAGIC_NUMBER) throw new IllegalArgumentException("非法请求");

    // 2. 读取版本和序列化器编号
    byte codec = in.readByte();
    byte version = (byte) (codec >> 4);        // 高 4 位
    byte serializerType = (byte) (codec & 0x0F); // 低 4 位

    // 3. 读取消息体长度
    int length = in.readInt();

    // 4. 根据长度读取消息体
    byte[] bodyBytes = new byte[length];
    in.readBytes(bodyBytes);

    // 5. 反序列化
    Object body = serializer.deserialize(bodyBytes, messageType);
    return RpcMessage.builder().data(body).codec(codec)....build();
}
```

## 四、TCP 粘包/拆包问题

### 4.1 什么是粘包/拆包

TCP 是**流式协议**，没有消息边界。发送方连续 write 两次，接收方可能：

```
发送：  [Msg1: 100B]  [Msg2: 200B]
         |               |
接收可能：                   
情况1:  [Msg1][Msg2]       → 粘包（两条消息粘在一起到了）
情况2:  [Msg1前半][Msg1后半+Msg2] → 拆包（一条消息被拆分）
情况3:  [Msg1][Msg2前半][Msg2后半]
```

### 4.2 解决方案

本项目的解决方式：**自定义协议中的 Length 字段 + Netty 的 LengthFieldBasedFrameDecoder**

```java
// RpcMessageFrameDecoder.java
public class RpcMessageFrameDecoder extends LengthFieldBasedFrameDecoder {
    public RpcMessageFrameDecoder() {
        super(
            MAX_FRAME_LENGTH,           // 最大帧长度
            5,                          // length 字段偏移量 = Magic(4) + Version(1) = 5
            4,                          // length 字段长度（4 字节）
            -9,                         // 调整量
            0                           // 初始跳过字节数
        );
    }
}
```

参数解读：
- **lengthFieldOffset = 5**：Body Length 字段从第 5 个字节开始（0-3 是魔数，4 是版本+序列化器）
- **lengthFieldLength = 4**：Body Length 本身占 4 个字节
- **lengthAdjustment = -9**：修正值，使得解码后只保留 Body Data 部分

**面试话术**：「我通过在协议头部设计了 4 字节的 Body Length 字段，配合 Netty 的 LengthFieldBasedFrameDecoder，根据 Length 动态确定每帧的边界，从而解决了 TCP 粘包/拆包问题。」

## 五、连接复用机制

### 5.1 为什么需要连接复用

如果每次 RPC 调用都 `new connect → 发送 → close`：
- TCP 三次握手耗时 ~1.5 RTT
- TCP 四次挥手有 TIME_WAIT
- 频繁创建/销毁连接浪费资源

### 5.2 本项目实现

```java
// ChannelProvider.java — 核心逻辑
public Channel get(InetSocketAddress address) {
    String key = address.toString();
    Channel channel = CHANNEL_MAP.get(key);
    // 连接不存在或已断开 → 创建新连接
    if (channel == null || !channel.isActive()) {
        channel = bootstrap.connect(address).sync().channel();
        CHANNEL_MAP.put(key, channel);
    }
    return channel;
}
```

与 CompletableFuture 的协作：
- 多个请求可以**复用同一个 Channel** 发送
- 通过 requestId 区分不同的请求和响应
- Netty Channel 的 writeAndFlush 是**线程安全**的，可以多线程并发发送

## 六、NettyRpcServer 完整启动流程

```java
// NettyRpcServer.java
public void start() {
    // 1. 清理旧资源（如果有）
    CustomShutdownHook.clearAll();
    
    // 2. 创建 EventLoopGroup
    bossGroup = new NioEventLoopGroup(1);           // 1 个 boss 线程
    workerGroup = new NioEventLoopGroup();           // 默认 CPU 核数×2
    
    try {
        ServerBootstrap bootstrap = new ServerBootstrap();
        bootstrap.group(bossGroup, workerGroup)
                 .channel(NioServerSocketChannel.class)
                 // TCP 参数调优
                 .childOption(ChannelOption.TCP_NODELAY, true)    // 关闭 Nagle 算法
                 .childOption(ChannelOption.SO_KEEPALIVE, true)   // TCP 保活
                 .childHandler(new ChannelInitializer<SocketChannel>() {
                     @Override
                     protected void initChannel(SocketChannel ch) {
                         // 心跳：30 秒没有读 → 发心跳 → 3 次没响应 → 断开
                         ch.pipeline().addLast(new IdleStateHandler(30, 0, 0, TimeUnit.SECONDS));
                         ch.pipeline().addLast(new RpcMessageFrameDecoder());
                         ch.pipeline().addLast(new RpcMessageDecoder());
                         ch.pipeline().addLast(new RpcMessageEncoder());
                         ch.pipeline().addLast(new NettyRpcServerHandler());
                     }
                 });
        
        // 3. 绑定端口并启动
        ChannelFuture future = bootstrap.bind(port).sync();
        future.channel().closeFuture().sync();
    } finally {
        // 4. 优雅关闭
        bossGroup.shutdownGracefully();
        workerGroup.shutdownGracefully();
    }
}
```

## 七、本章面试要点

1. **为什么用 Netty 而不是传统 Socket？** → NIO vs BIO，Reactor 模型，零拷贝，高并发支撑
2. **Reactor 模型是什么？Boss 和 Worker 怎么分工？** → Boss 负责 accept，Worker 负责读写
3. **自定义协议怎么设计的？** → 魔数 + 版本 + 序列化器编号 + 长度 + 体，头部仅 9 字节
4. **粘包/拆包怎么解决？** → 协议中的 Length 字段 + Netty 的 LengthFieldBasedFrameDecoder
5. **ChannelPipeline 的顺序为什么重要？** → 帧解码必须最先，否则后续 Handler 收到的是不完整的数据
6. **连接复用的实现和好处？** → ChannelProvider 维护 Channel 池，避免频繁建连和断连开销
7. **TCP_NODELAY 是什么？** → 关闭 Nagle 算法，小包不等待合并，减少延迟
