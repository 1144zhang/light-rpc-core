# RPC 基础理论

## 一、什么是 RPC

**RPC（Remote Procedure Call，远程过程调用）**，一句话解释：

> 像调用本地方法一样，调用远程服务器上的方法。

举个例子，你在本地写代码：

```java
// 本地调用
HelloService service = new HelloServiceImpl();
String result = service.hello(new Hello("张三"));
```

这是本地调用，对象在同一个 JVM 里。

RPC 的目标是让你在写代码时，**感受不到这是远程调用**：

```java
// RPC 调用 — 实际执行在远程服务器上，但代码写法跟本地一样
@RpcReference
private HelloService helloService;  // 这是个远程服务

String result = helloService.hello(new Hello("张三"));  // 实际走了网络
```

## 二、为什么需要 RPC

### 2.1 单体架构 → 微服务架构

```
┌─────────────────────────┐       ┌──────────┐  ┌──────────┐  ┌──────────┐
│  单体应用                 │  →    │ 用户服务  │  │ 订单服务  │  │ 商品服务  │
│  用户+订单+商品+支付      │       │          │  │          │  │          │
│  全部在一个工程里         │       └──────────┘  └──────────┘  └──────────┘
└─────────────────────────┘              ▲              ▲              ▲
                                         └──────────────┴──────────────┘
                                                 不同的进程/机器
                                                 RPC 负责它们之间的通信
```

拆成微服务后，原本在同一个进程里的方法调用，变成了跨进程甚至跨机器的调用。RPC 就是用来解决这个问题的。

### 2.2 RPC vs HTTP

面试高频题：「你们的 RPC 为什么不用 HTTP？」

| 对比维度 | RPC | HTTP |
|----------|-----|------|
| 协议层 | 基于 TCP，自定义协议 | 基于 TCP，HTTP 协议 |
| 协议头部 | 极小（几十字节） | 较大（每次请求带完整 HTTP 头） |
| 序列化 | 高效二进制（Protobuf/Kryo） | 通常 JSON（文本，体积大） |
| 性能 | 高（连接池复用 + 二进制） | 较低（每次可能重新建连） |
| 服务治理 | 自带（注册中心、负载均衡、熔断） | 需额外组件 |
| 跨语言 | 部分框架支持（gRPC） | 天然支持 |
| 适用场景 | 微服务内部调用 | 对外 API、跨语言场景 |

**一句话总结**：RPC 追求**高性能和强类型**，HTTP 追求**通用性和简单**。

## 三、RPC 的核心流程

一次 RPC 调用，完整经历以下 6 个步骤：

```
┌──────────────────────────────────────────────────────────────────┐
│  Client 端                                                       │
│                                                                  │
│  ① 客户端调用接口方法                                              │
│      │                                                            │
│      ▼                                                            │
│  ② 动态代理拦截 → 封装成 RpcRequest（类名、方法名、参数、requestId）│
│      │                                                            │
│      ▼                                                            │
│  ③ 序列化 → 字节数组                                               │
│      │                                                            │
│      ▼                                                            │
│  ④ 网络传输（Netty Client → TCP → Netty Server）                    │
│                                                                  │
├──────────────────────────────────────────────────────────────────┤
│  Server 端                                                       │
│                                                                  │
│  ⑤ 反序列化 → RpcRequest 对象                                      │
│      │                                                            │
│      ▼                                                            │
│  ⑥ 反射调用目标方法 → 得到结果                                      │
│      │                                                            │
│      ▼                                                            │
│  ⑦ 序列化返回结果 → 网络传输 → Client 反序列化 → 返回给调用方        │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 对应到本项目的代码

```java
// 步骤 ②：RpcClientProxy.java
RpcRequest rpcRequest = RpcRequest.builder()
    .interfaceName(method.getDeclaringClass().getName())
    .methodName(method.getName())
    .parameters(args)
    .paramTypes(method.getParameterTypes())
    .requestId(UUID.randomUUID().toString())
    .group(group)
    .version(version)
    .build();

// 步骤 ④：NettyRpcClient.java → Channel.writeAndFlush()
channel.writeAndFlush(rpcRequest).addListener(future -> { ... });

// 步骤 ⑥：RpcRequestHandler.java
Object result = method.invoke(service, rpcRequest.getParameters());
```

## 四、RPC 框架的核心组件

任何一个 RPC 框架，都包含以下核心模块：

```
                    ┌─────────────┐
                    │  Client 端  │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼─────┐ ┌───▼────┐ ┌────▼─────┐
        │ 动态代理    │ │ 序列化  │ │ 注册中心  │
        │(屏蔽远程)  │ │(对象↔字节)│ │(服务发现)  │
        └─────┬─────┘ └───┬────┘ └────┬─────┘
              │            │            │
              └────────────┼────────────┘
                           │
                    ┌──────▼──────┐
                    │   网络传输    │
                    │  (Netty NIO) │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼─────┐ ┌───▼────┐ ┌────▼─────┐
        │ 负载均衡   │ │连接管理 │ │ 心跳检测  │
        │(选哪个节点)│ │(Channel池)│ │(保活)    │
        └───────────┘ └────────┘ └──────────┘
```

| 组件 | 职责 | 本项目对应 |
|------|------|-----------|
| **动态代理** | 让调用方感知不到远程细节 | `RpcClientProxy` |
| **序列化** | 对象 ↔ 字节流转换 | `KryoSerializer` 等 |
| **注册中心** | 服务地址的注册与发现 | Zookeeper + Curator |
| **网络传输** | 数据的实际发送与接收 | Netty（Rector 模型） |
| **负载均衡** | 从多个节点中选一个 | `ConsistentHashLoadBalance` |
| **连接管理** | 复用已建立的连接 | `ChannelProvider` |
| **心跳检测** | 检测连接是否存活 | `IdleStateHandler` |

## 五、RPC 调用的时序细节

以本项目为例，一次完整调用的时序：

```
Client(调用方)              Zookeeper              Server(服务方)
     │                          │                      │
     │  1. 启动时发现服务        │                      │
     │─────────────────────────>│                      │
     │  2. 返回服务地址列表      │                      │
     │<─────────────────────────│                      │
     │                          │    3. 注册服务        │
     │                          │<─────────────────────│
     │                          │                      │
     │  4. 从地址列表选一个节点   │                      │
     │  (负载均衡)               │                      │
     │                          │                      │
     │  5. 建立/复用 Channel    │                      │
     │────────────────────────────────────────────────>│
     │                          │                      │
     │  6. 发送 RpcRequest      │                      │
     │  (序列化后的字节流)        │                      │
     │────────────────────────────────────────────────>│
     │                          │                      │
     │                          │   7. Server 接收      │
     │                          │   反序列化 → 反射调用  │
     │                          │                      │
     │  8. 返回 RpcResponse     │                      │
     │<────────────────────────────────────────────────│
     │                          │                      │
     │  9. 反序列化 → 返回给调用方│                      │
```

## 六、主流 RPC 框架对比

面试时常问「你的框架和 Dubbo/gRPC 有什么区别？」，提前了解：

| 框架 | 语言 | 序列化 | 注册中心 | 特点 |
|------|------|--------|----------|------|
| **Dubbo** | Java | Hessian2/Protobuf | ZK/Nacos/Redis | 阿里开源，国内最流行 |
| **gRPC** | 多语言 | Protobuf | 可插拔 | Google 开源，HTTP/2 |
| **Thrift** | 多语言 | 自带二进制 | 可插拔 | Facebook 开源 |
| **Motan** | Java | JSON/Hessian2 | ZK/Consul | 微博开源，轻量 |
| **light-rpc-core** | Java | Kryo/Protostuff/Hessian | ZK | 学习项目，适合理解原理 |
