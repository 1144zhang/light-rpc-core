# 注册中心详解

## 一、为什么需要注册中心

### 1.1 没有注册中心的世界

```
Client ──直接调用──> Server (192.168.1.10:8080)
```

问题：
- Server 地址**硬编码**在 Client 代码里
- Server 扩容（加一台 192.168.1.11）→ 要改 Client 代码
- Server 挂了 → Client 不知道，继续发请求 → 全部失败
- Server 迁移 IP → 又要改代码

### 1.2 有了注册中心

```
┌──────────┐     注册      ┌─────────────┐     注册     ┌──────────┐
│ Server A │ ───────────> │ Zookeeper   │ <─────────── │ Server B │
│ :8080    │              │ 注册中心      │              │ :8081    │
└──────────┘              └──────┬──────┘              └──────────┘
                                 │ 服务发现
                          ┌──────▼──────┐
                          │   Client    │  获取列表: [A:8080, B:8081]
                          │             │  选一个调用
                          └─────────────┘
```

优势：
- **动态扩容**：新节点注册即可，Client 自动发现
- **故障摘除**：节点宕机，注册中心自动移除
- **负载均衡**：Client 从多个节点中选一个

## 二、为什么选 Zookeeper

### 2.1 ZK 的核心特性

| 特性 | 说明 | 对 RPC 的意义 |
|------|------|--------------|
| **临时节点** | 客户端断开后自动删除 | 服务端宕机 → 自动摘除 |
| **Watch 机制** | 监听节点变化，主动通知 | 服务列表变更 → 秒级感知 |
| **强一致性 (CP)** | 数据保证一致 | 服务列表不会读到脏数据 |
| **树形结构** | `/rpc/serviceName/provider1` | 天然适合组织服务地址 |

### 2.2 ZK vs Nacos vs Eureka

| 对比维度 | Zookeeper | Nacos | Eureka |
|----------|-----------|-------|--------|
| CAP 模型 | CP | AP/CP 可切换 | AP |
| 一致性 | 强一致 | 可配 | 最终一致 |
| 健康检查 | 临时节点 + 心跳 | 主动上报 + 被动探测 | 心跳续约 |
| 管理界面 | 无（需额外工具） | 有 | 有 |
| 学习成本 | 中 | 中 | 低 |

**为什么本项目用 ZK**：
1. 临时节点天然支持故障摘除
2. Watch 机制实现高效变更通知
3. 社区成熟，Curator 客户端封装完善

## 三、服务注册实现

### 3.1 注册路径设计

```
/light-rpc-core                            ← 根节点
└── /com.zhang.HelloService                ← 服务名（接口全限定名）
    └── /group1                            ← 服务分组
        └── /version1.0                     ← 服务版本
            ├── /192.168.1.10:8080         ← 服务地址（临时节点）
            ├── /192.168.1.11:8081         ← 服务地址（临时节点）
            └── /192.168.1.12:8082         ← 服务地址（临时节点）
```

路径设计亮点：
- 服务名 + 分组 + 版本三级定位，支持灰度发布和多版本共存
- 地址节点是**临时节点**，Server 断开后自动删除

### 3.2 核心代码

```java
// ZkServiceRegistryImpl.java
public void registerService(String serviceName, InetSocketAddress address) {
    // 构建注册路径：/light-rpc-core/com.zhang.HelloService/group1/version1.0/192.168.1.10:8080
    String servicePath = CuratorUtils.ZK_REGISTER_ROOT_PATH + "/" + serviceName;
    String path = servicePath + "/" + address.toString();

    // 创建持久节点（父节点）+ 临时节点（地址节点）
    CuratorFramework zkClient = CuratorUtils.getZkClient();
    zkClient.create()
            .creatingParentsIfNeeded()                    // 父节点不存在则自动创建
            .withMode(CreateMode.EPHEMERAL)               // ★ 临时节点：断连自动删除
            .forPath(path);
}
```

**关键理解**：
- `creatingParentsIfNeeded()`：第一次注册时自动创建 `/rpc/serviceName/group/version` 路径
- `CreateMode.EPHEMERAL`：如果 Server 宕机，ZK 会话断开，这个地址节点自动删除

## 四、服务发现实现

### 4.1 核心类

```java
// ZkServiceDiscoveryImpl.java
public class ZkServiceDiscoveryImpl implements ServiceDiscovery {
    // ★ 本地缓存：避免每次都查 ZK
    private final Map<String, List<String>> SERVICE_ADDRESS_MAP = new ConcurrentHashMap<>();

    public List<String> lookupService(String serviceName) {
        // 1. 先查本地缓存（有 Watch 机制保证新鲜度）
        if (SERVICE_ADDRESS_MAP.containsKey(serviceName)) {
            return SERVICE_ADDRESS_MAP.get(serviceName);
        }
        // 2. 查 ZK
        return getServiceAddressList(serviceName);
    }
}
```

### 4.2 Watch 机制：被动感知服务上下线

```java
private List<String> getServiceAddressList(String serviceName) {
    String servicePath = CuratorUtils.ZK_REGISTER_ROOT_PATH + "/" + serviceName;

    // 获取所有子节点
    List<String> children = zkClient.getChildren()
            .usingWatcher(new CuratorWatcher() {   // ★ 注册 Watcher
                @Override
                public void process(WatchedEvent event) {
                    // 子节点变化时触发（服务上线/下线）
                    if (event.getType() == Watcher.Event.EventType.NodeChildrenChanged) {
                        // ★ 清除缓存，下次请求时重新加载
                        SERVICE_ADDRESS_MAP.remove(serviceName);
                    }
                }
            })
            .forPath(servicePath);

    // 存入缓存
    SERVICE_ADDRESS_MAP.put(serviceName, children);
    return children;
}
```

**工作流程**：

```
1. Client 第一次查 → ZK 返回 [A, B] → 缓存到本地
2. Server C 注册（新节点上线）
3. ZK 通知 Client → Watch 触发 → 清除缓存
4. Client 下次查 → 缓存未命中 → 重新查 ZK → 返回 [A, B, C]
5. Server B 宕机（临时节点自动删除）
6. ZK 通知 Client → Watch 触发 → 清除缓存 → 重新加载 → [A, C]
```

**面试重点**：缓存 + Watch 机制既保证了**性能**（避免每次查 ZK），又保证了**实时性**（变更秒级感知）。

## 五、Curator 框架

本项目使用 Apache Curator 操作 ZK，而不是原生的 ZooKeeper 客户端。

### 5.1 为什么用 Curator

| 原生 ZK Client | Curator |
|---------------|---------|
| Watcher 一次性（触发后需重新注册） | 自动重新注册 Watcher |
| 连接断开需手动重连 | 自动重连 + 重试策略 |
| API 复杂，需大量样板代码 | Fluent API，链式调用 |
| 没有分布式锁等高级功能 | 内置分布式锁、选举等 |

### 5.2 本项目中的 Curator 工具类

```java
// CuratorUtils.java
public class CuratorUtils {
    private static CuratorFramework zkClient;

    public static CuratorFramework getZkClient() {
        if (zkClient == null) {
            // ★ 重试策略：每 3 秒重试一次，最多重试 3 次
            RetryPolicy retryPolicy = new RetryNTimes(3, 3000);
            zkClient = CuratorFrameworkFactory.builder()
                    .connectString("127.0.0.1:2181")
                    .retryPolicy(retryPolicy)
                    .sessionTimeoutMs(60000)
                    .connectionTimeoutMs(15000)
                    .build();
            zkClient.start();
        }
        return zkClient;
    }
}
```

## 六、CAP 理论与注册中心选型

面试高频题：「注册中心应该选 AP 还是 CP？」

### 6.1 CAP 理论

| 字母 | 含义 | 说明 |
|------|------|------|
| **C** (Consistency) | 一致性 | 所有节点同一时刻看到相同数据 |
| **A** (Availability) | 可用性 | 每次请求都能收到非错的响应 |
| **P** (Partition Tolerance) | 分区容错 | 网络分区时系统仍能工作 |

CAP 定理：分布式系统最多只能同时满足其中两个。

### 6.2 ZK vs Eureka 的设计哲学

**Zookeeper（CP）**：
- 追求一致性
- 当 Leader 宕机，集群进入选举状态时，整个 ZK 不可用
- 但此时**服务调用不应受影响**（已有缓存的服务列表）

**Eureka（AP）**：
- 追求可用性
- 每个节点平等，部分节点挂掉不影响其他节点
- 可能返回过期的服务信息（宁可给错的，也不不给）

### 6.3 面试标准回答

> 「注册中心选型取决于业务对一致性和可用性的权衡。对于绝大多数微服务场景，**AP 模型更实用**——因为注册中心短暂不可用不应该阻止服务调用。但在本项目中使用 ZK 是为了学习和理解 CP 模型下注册中心的实现方式。实际生产环境中，Nacos 因为可以灵活切换 AP/CP，是比较均衡的选择。」

## 七、本章面试要点

1. **为什么需要注册中心？** → 解耦服务地址，支持动态扩缩容和故障摘除
2. **ZK 的临时节点和 Watch 机制怎么用的？** → 临时节点实现故障自动摘除，Watch 实现缓存实时更新
3. **为什么用 Curator 而不是原生 ZK 客户端？** → 自动重连、Watcher 自动重注册、Fluent API
4. **本地缓存的作用？** → 减少 ZK 压力，配合 Watch 保证实时性
5. **注册中心选 CP 还是 AP？** → 多数场景选 AP（可用性优于一致性），Nacos 两者兼顾
6. **如果 ZK 挂了，RPC 还能调用吗？** → 能！因为 Client 有本地缓存的服务列表，只是无法感知新节点上下线
